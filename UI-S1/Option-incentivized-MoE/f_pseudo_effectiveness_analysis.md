## 1. 背景与定义

### f_pseudo 奖励塑形

f_pseudo 基于 **eigenfunction** 的伪奖励信号：

```
f_pseudo(t) = f(s_t) - f(s_{t+1})
```

- `f(s)`: 由 per-app 的 eigenfunction network (f_net) 预测的状态值
- 当 agent 从高 f 区域移到低 f 区域（crossing connectivity bottleneck）时，`f_pseudo > 0`，给予正奖励
- 总奖励: `r_total = r_action_match + λ × f_pseudo`

### 数据来源

| 文件 | 说明 |
|------|------|
| `outputs/f_pseudo/f_pseudo_map.json` | 预计算的 f_pseudo 值（按 execution_id × step_id） |
| `outputs/f_pseudo/statistics.json` | f_pseudo 全局统计 |
| `outputs/fnet/gui360/{app}/f_values.npz` | per-app eigenfunction 的 f 值 |
| `outputs/transitions/gui360_full/transitions.jsonl` | 状态转移数据（含 src_hash, dst_hash） |
| `datasets/GUI-360/rl_data/gui360_train.jsonl` | RL 训练数据 |

## 3. Off-by-one 对齐 Bug 及修复

### 3.1 根因

| 组件 | Step 索引方式 |
|------|:-----------:|
| `transitions.jsonl` → `f_pseudo_map` | **1-indexed** (step_id 从 1 开始) |
| `x/data/agent/base.py` line 82: `state['step_id'] = si` | **0-indexed** (si 从 0 开始) |
| `f_pseudo_dapo.py` (修复前) | 直接用 `step_id` 查询 → **100% 错位** |

### 3.2 错位示例

以轨迹 `excel_1_105` 为例（5 个 RL 步骤，4 个 f_pseudo 值）:

```
f_pseudo_map 键值: {'1': -1.548, '2': -0.129, '3': 0.0, '4': 0.0}

修复前:                                    修复后:
si=0 → lookup "0" → MISS (0.0)            si=0 → lookup "1" → -1.548 ✓
si=1 → lookup "1" → -1.548 (错位!)        si=1 → lookup "2" → -0.129 ✓
si=2 → lookup "2" → -0.129 (错位!)        si=2 → lookup "3" → 0.0   ✓
si=3 → lookup "3" → 0.0                   si=3 → lookup "4" → 0.0   ✓
si=4 → lookup "4" → 0.0 (不应有值!)       si=4 → 最后一步, 跳过     ✓
```

### 3.3 影响量化

| 指标 | 数量 | 占比 |
|------|:----:|:----:|
| RL 数据总步数 | 104,855 | 100% |
| 第一步 MISS（si=0 查 "0"） | 13,237 | 12.6% |
| 中间步错位 | 78,381 | 74.8% |
| 最后步获得无关值 | 13,237 | 12.6% |

中间步错位的影响:

| 指标 | 值 |
|------|:---:|
| 恰好正确的（碰巧） | 36.7% |
| 平均 |错位值 - 正确值| | 0.0687 |
| 中位数 |错位值 - 正确值| | 0.0090 |
| 最大 |错位值 - 正确值| | 6.387 |
| 差异 > 0.01 的比例 | 49.1% |
| 差异 > 0.05 的比例 | 23.8% |

### 3.4 步数对齐

f_pseudo 始终有 **N-1** 个条目对应 **N** 个 RL 步骤（因为 f_pseudo(t) = f(s_t) - f(s_{t+1}) 产生 N-1 个转移值）。

### 3.5 修复内容

文件: `verl/workers/reward_manager/f_pseudo_dapo.py` 第 178-192 行

**修复前:**
```python
f_pseudo = 0.0
if self.f_pseudo_map and self.f_pseudo_lambda != 0.0:
    execution_id = data_item.non_tensor_batch.get("execution_id", "")
    step_id = data_item.non_tensor_batch.get("step_id", "")
    if execution_id:
        f_pseudo = self._lookup_f_pseudo(execution_id, step_id)
```

**修复后:**
```python
# f_pseudo_map uses 1-indexed step keys ("1","2",...) from transitions.jsonl,
# while the RL trainer sets step_id = si (0-indexed).
# Fix: map si → si+1 for lookup, and skip the last step (no next state).
f_pseudo = 0.0
if self.f_pseudo_map and self.f_pseudo_lambda != 0.0:
    execution_id = data_item.non_tensor_batch.get("execution_id", "")
    step_id = data_item.non_tensor_batch.get("step_id", -1)
    num_steps = ground_truth.get("num_steps", 0) if isinstance(ground_truth, dict) else 0
    if execution_id and step_id >= 0:
        # Last step has no next state → no f_pseudo
        if num_steps > 0 and step_id >= num_steps - 1:
            f_pseudo = 0.0
        else:
            f_pseudo = self._lookup_f_pseudo(execution_id, step_id + 1)
```

## 4. 信号稀疏性诊断（Q1-Q8）

### Q1: 为什么 54.6% 的步骤 f_pseudo = 0？

| 类别 | 数量 | 占比 |
|------|:----:|:----:|
| 总步数 | 91,618 | 100% |
| **非零 f_pseudo** | **41,554** | **45.4%** |

> **结论**: 几乎所有零值都是因为**自环**——action 未改变 screenshot hash。仅 17 个步骤是不同状态但 f 值碰巧相等。

## 7. 根本问题分析

### 7.1 Eigenfunction 学到的是结构而非方向

当前 eigenfunction 学习的是转移图的 **Fiedler vector**（第二小特征向量），它描述图的 **connectivity bottleneck** 但**没有方向性**:

```
Fiedler vector 性质:
- 将图分成两半（一正一负）
- 过零点对应图的"瓶颈切割"
- 但哪半是"起点"、哪半是"终点" 是任意的
- 因此 f(s_start) ≈ f(s_end) → 累积 f_pseudo ≈ 0
```

### 7.2 状态表示瓶颈

Screenshot hash 作为状态标识的粒度太粗:

- `click` 操作（如点击菜单、选择单元格）可能改变 UI 状态但不改变整体截图 hash
- `type` 操作（如输入文字）改变的是小区域像素，hash 可能不变
- 结果: 54.6% 的真实状态转移被误判为自环，f_pseudo = 0

### 7.3 信号退化路径

```
长轨迹 → 更多步骤 → 更多自环零值 (72.6%)
          → 更多正负振荡 → 低符号一致性 (68.9%)
          → 累积 ≈ 0 → 无净方向性
          → f_pseudo 退化为噪声
```

## 8. 验证 f_pseudo 有效性的方法与建议

### 方法 1: 离线 Ablation 实验（最直接, 推荐首先执行）

直接对比有/无 f_pseudo 的训练曲线:

```yaml
# 实验 A: baseline (无 f_pseudo)
f_pseudo_lambda: 0.0

# 实验 B: 当前设定
f_pseudo_lambda: 0.1

# 实验 C: 更弱信号
f_pseudo_lambda: 0.05
```

评估指标（按 Short/Medium/Long 分组）:
- Trajectory Success Rate (TSR)
- Step-level Success Rate
- Average Progress Rate

> 如果 A ≈ B ≈ C，则 f_pseudo 当前形式**确认无效**。

### 方法 2: 离线 Reward Correlation 分析

训练前即可验证——检查 f_pseudo bonus 与 trajectory outcome 的相关性:

```python
# 对每条轨迹计算:
# 1. cumulative_f_pseudo = sum(f_pseudo for each step)
# 2. trajectory_success = 1 or 0
# 3. correlation(cumulative_f_pseudo, trajectory_success)
#
# 已知: cumulative ≈ 0 → 相关性很可能接近 0
```

### 方法 3: 改进状态表示（解决 54.6% 自环问题）

| 方案 | 复杂度 | 预期效果 |
|------|:------:|---------|
| **视觉 embedding hash**（Qwen-VL encoder CLS token → LSH） | 中 | 区分细微 UI 变化，大幅减少自环 |
| **结构化状态**（DOM / a11y tree hash） | 低 | 文本变化可检测，但需要数据支持 |
| **Action-augmented state**（state = screenshot + last_action） | 低 | 简单缓解但不从根本解决 |

### 方法 4: 替换为有方向性的 f(s)（解决累积和≈0 的根本问题）

当前 eigenfunction 无方向性，替代方案:

| 方案 | 说明 | 优势 |
|------|------|------|
| **Goal-conditioned Value Function** | 训 V(s, g) 预测从 s 到 goal g 的剩余步数 | 保证 f(start) > f(end), 有净方向信号 |
| **Progress Estimator** | 训 p(s) ∈ [0, 1] 预测任务完成进度 | 简单直接, 保证 f(s_0)≈0, f(s_final)≈1 |
| **Successor Representation** | 用 successor feature 学有方向性的状态表示 | 理论基础强, 但实现复杂 |

> Progress Estimator 实现最简单: 训一个 MLP 预测 `step_id / num_steps`，作为 f(s)。这保证累积 f_pseudo ≈ 1 > 0。

### 方法 5: Per-trajectory f_pseudo 归一化

不改 eigenfunction，而是对 f_pseudo 做 per-trajectory 标准化:

```python
# 对每条轨迹:
# 1. 计算 f_pseudo 序列: [f1, f2, ..., fn]
# 2. 归一化: f_norm = (f - mean) / (std + eps)
# 3. 只保留正值: f_bonus = max(f_norm, 0)
```

这不解决方向性问题，但能增强有信号步骤的区分度。

### 建议优先级

| 优先级 | 方法 | 工作量 | 预期效果 |
|:------:|------|:------:|---------|
| 1 | **Ablation 实验** (方法 1) | 低 (改 1 行 config) | 确认当前 f_pseudo 是否有效 |
| 2 | **Progress Estimator** (方法 4) | 中 (训新模型) | 如果 Ablation 无效，这是最直接的替代方案 |
| 3 | **改进状态表示** (方法 3) | 中-高 | 长期改善，减少自环 |
| 4 | **Per-traj 归一化** (方法 5) | 低 | 短期缓解，与其他方法可叠加 |
