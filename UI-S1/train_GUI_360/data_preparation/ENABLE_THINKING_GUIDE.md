# Enable Thinking in GUI-360 SFT Training

This guide explains how to include reasoning/thinking in the GUI-360 SFT training data to improve model performance.

---

## Problem Summary

Current SFT training causes **catastrophic forgetting** because:

| Issue | Current State | Impact |
|-------|--------------|--------|
| Training data format | Pure `<tool_call>` JSON only | Model loses reasoning ability |
| Output mismatch | Training = JSON, Base model = Reasoning + JSON | Corrupted hybrid output |
| `thought` field | Lost during processing | No reasoning supervision |
| `enable_thinking` | Disabled in training | No thinking tags |

**Result**: Full accuracy drops from 37.01% (base) to 5.81% (step 539)

---

## Solution: Two Approaches

### Approach 1: Include `thought` in Assistant Response (Recommended)

Add the raw `thought` field as reasoning prefix before the action:

```python
# Current format (problematic)
assistant_response = '<tool_call>\n{"function": "click", ...}\n</tool_call>'

# New format with thinking
thought = step.get('thought', '')
assistant_response = f"Reasoning: {thought}\n\n<tool_call>\n{json.dumps(action)}\n</tool_call>"
```

**Pros:**
- Uses existing data (raw JSONL has `thought` field)
- Matches base model's natural output style
- No tokenizer changes needed

**Cons:**
- Need to regenerate parquet from raw JSONL files
- `thought` quality varies

### Approach 2: Use Qwen's Native `enable_thinking` Mode

Add `enable_thinking` column to parquet data and let the tokenizer handle it:

```python
# In parquet data
df['enable_thinking'] = True

# Tokenizer will add <think>...</think> tags automatically
tokenizer.apply_chat_template(messages, enable_thinking=True)
```

**Pros:**
- Uses Qwen's built-in thinking format
- More structured thinking tags

**Cons:**
- Need to provide thinking content separately
- Requires understanding Qwen's thinking format

## Implementation Details

### Current Data Pipeline

```
Raw JSONL (has thought)           Processed JSON (lost thought)        Parquet (no thinking)
├── step.thought ✅         →     ├── conversation                →    ├── messages
├── step.observation ✅           │   └── assistant: <tool_call>       └── (no enable_thinking)
└── step.action                   └── (thought LOST!)
```

### Required Changes

#### Option A: Regenerate with Reasoning Prefix

1. **Create new data prep script** that reads from raw JSONL
2. **Include `thought` field** in assistant response
3. **Generate new parquet** with reasoning format

```python
# New assistant response format
def create_assistant_response(step):
    thought = step.get('thought', '')
    action = step.get('action', {})

    response = ""
    if thought:
        response += f"Reasoning: {thought}\n\n"

    response += f"<tool_call>\n{json.dumps(action, indent=2)}\n</tool_call>"
    return response
```

#### Option B: Enable Qwen Thinking Mode

1. **Add `enable_thinking` column** to parquet (set to `True`)
2. **Add `thinking_content` to messages** (optional, for structured thinking)
3. **Config already supports it**: `enable_thinking_key: enable_thinking`

```yaml
# Current config (already correct)
data:
  multiturn:
    enable: true
    messages_key: messages
    enable_thinking_key: enable_thinking  # ← Already configured!
```

From raw JSONL files, the `thought` field contains:

```json
{

                template.",
    "observation": "I see the PowerPoint window with the business plan template open...",
    "subtask": "Click on the File Tab",
    "action": {...}
  }
}
```

**Quality**: High-quality reasoning generated by GPT-4 during data collection

### Fields Available in Raw Data

| Field | Description | Use Case |
|-------|-------------|----------|
| `thought` | Reasoning about next action | Primary thinking content |
| `observation` | Description of current screen | Context for reasoning |
| `subtask` | High-level goal | Task decomposition |
| `action` | The actual action | Output target |

## Recommended Implementation Plan

### Step 1: Create New Data Preparation Script

Create `prepare_gui360_sft_with_thinking.py`:

```python
"""
Convert GUI-360 raw JSONL to parquet with thinking/reasoning included.
"""

def process_step(data: dict) -> dict:
    step = data.get('step', {})

    # Extract reasoning components
    thought = step.get('thought', '')
    observation = step.get('observation', '')

    # Extract action
    action = step.get('action', {})
    status = step.get('status', 'CONTINUE')

    # Build action JSON
    action_json = {
        "function": action.get('function', ''),
        "args": {
            "coordinate": [action.get('coordinate_x'), action.get('coordinate_y')],
            **action.get('args', {})
        },
        "status": status
    }

    # Build assistant response WITH reasoning
    assistant_response = ""
    if thought:
        assistant_response += f"Reasoning: {thought}\n\n"
    assistant_response += f"<tool_call>\n{json.dumps(action_json, indent=2)}\n</tool_call>"

    return {
        "messages": [...],  # user message + assistant response
        "enable_thinking": True  # Enable tokenizer thinking mode
    }
```

### Step 2: Update Config (Optional)

The config already supports `enable_thinking`. No changes needed if parquet has the column.

### Step 3: Regenerate Training Data

```bash
python train_GUI_360/data_preparation/prepare_gui360_sft_with_thinking.py \
    --input-dir datasets/GUI-360/train/data \
    --output train_GUI_360/data/gui360_train_sft_with_thinking.parquet \
    --image-base-dir datasets/GUI-360/train
```

### Step 4: Update Training Config

```yaml
data:
  train_files: /path/to/gui360_train_sft_with_thinking.parquet
```

## Expected Impact

Based on root cause analysis:

| Metric | Current (No Thinking) | Expected (With Thinking) |
|--------|----------------------|--------------------------|
| Format retention | Degrades to 7.9% | Should maintain >50% |
| FINISH over-prediction | 5x ground truth | Normal distribution |
| Full accuracy | Drops to 5.81% | Should stay >25% |

### Why This Should Work

1. **Format Alignment**: Training format matches base model's natural output (reasoning + action)
2. **Reasoning Supervision**: Model learns to reason before acting
3. **Output Stability**: Less mode collapse to descriptions
4. **Better Generalization**: Explicit reasoning helps transfer

## Alternative: Qwen Thinking Tags

If using Qwen's native thinking format:

```python
# Message with thinking
messages = [
    {"role": "user", "content": [...]},
    {"role": "assistant", "content": "<think>reasoning here</think>\n<tool_call>...</tool_call>"}
]

# Or let tokenizer add tags
df['enable_thinking'] = True
```

The tokenizer will add appropriate thinking markers when `enable_thinking=True`.
